Part 1 — The Architecture Question

Is “CLI-first, TUI calls the CLI” a good model?

Short answer: the instinct is right, one mechanism is wrong. Adopt CLI-first as a contract. Don't make the TUI shell out to the CLI binary.

Keep — Good idea

CLI-first as a principle

Every capability headless, scriptable, and testable first. The TUI adds presentation only, never logic. This is the lazygit / k9s philosophy and it's the right north star.

Keep — Good idea

“Two products, one package”

That's already the shape: a UI-agnostic ScannerService with two front-ends. Lean into it — give both equal feature parity and ship them as peers.

Change — Don't do this

TUI spawning the CLI binary

Shelling out to edgewalker scan as a subprocess throws away the live typed progress stream, async cancellation, and the sudo/permission flow — and adds a JSON-parsing protocol you'd have to maintain.

Why not subprocess the CLI? (the honest trade-off)

The appeal of “TUI calls CLI” is guaranteed parity — they can't drift if one literally drives the other. But in this codebase the cost is real:

  • Live progress is in-process today. The TUI renders host_found / port_found / cred_found events as they happen via a typed progress_callback. A subprocess can only emit text — you'd have to invent and parse an NDJSON event protocol on stdout.
  • Cancellation & privilege flows. The TUI awaits a coroutine it can cancel, and can suspend() to forward a sudo prompt for nmap. Killing/signalling a child process and proxying its sudo prompt is far messier.
  • Cost & fragility. Spawning a fresh Python interpreter (textual, typer, pydantic, impacket…) per command is slow, and you'd serialize models → JSON → models on every call.

Recommended: CLI-first by contract

Introduce one thin use-case layer (e.g. engine.run_assessment(opts)) that the Typer CLI maps to 1:1. The TUI calls the same use-case functions in-process. Parity is guaranteed by construction because both adapters call the identical entry point — no subprocess required.

Concretely, this means moving the duplicated security-warning / override gating (currently in cli.py and dashboard.py) down into that shared layer, and letting the CLI's ScanController + the TUI screens become pure rendering.

Part 2 — The Visual Redesign

From single-log wizard → persistent dashboard

Today the main area is one RichLog that swaps between log / report / topology. The redesign adopts the persistent multi-panel paradigm (lazygit / k9s / btop): the risk grade, network summary, top findings, and device table are all visible at once, with the sidebar as a stable spatial anchor.

What changed & why

Design decisions

Persistent multi-panelGrade, network, findings, and devices are simultaneously visible. Builds spatial memory instead of hiding data behind view toggles.
Grade as heroThe A–F score is the product's whole value prop — it now gets a dedicated gauge with score bar, not a buried line of text.
Mnemonic keys, not arbitrary numbersscan, devices, findings replace [1]…[6]. Plus ↑↓ / Tab / / / ? — the TUI lingua franca.
Context footer + ? overlayFooter shows only what's actionable now; full keymap behind ?. Three-tier progressive disclosure.
Severity = label + colourNever colour alone. Each finding carries a text chip, so it survives NO_COLOR and colour-blind use (brand ships an Okabe-Ito skin already).
Quieter chromeDropped the blink “press ENTER” prompt and ASCII ------ dividers for hairline borders and background layering — depth without noise.
Brand-aligned paletteElectric Violet primary, Cyan strictly for accents/active state, Signal-Red reserved for critical. Mirrors the existing periphery skin.
Resilient at 80×24Sidebar collapses to icons first; panels stack below breakpoint. Constraint-based, never absolute positioning.
